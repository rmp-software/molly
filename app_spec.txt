<project_specification>
  <project_name>Molly — Gestão de Epilepsia Canina (Canine Epilepsy Management)</project_name>

  <overview>
    Molly is a private, single-user, mobile-first web app (PWA) for managing one dog's epilepsy.
    Named after the dog it was built for. It has two streams:

      1. Seizure log & trends — fast logging of seizure episodes (in a stressful moment, one-handed
         on a phone) plus a dashboard to spot whether frequency is changing, so the owner knows when
         to discuss a medication change with the vet.

      2. Medication & stock control — track continuous, OTC, and compounded medications, their dosing
         schedules, and current stock; project "days remaining" and "reorder by" dates; warn early
         enough to act (the compounded medication needs ~1 week lead time to reorder).

    This is a personal tool, not a clinical/medical device. It never prescribes — it surfaces
    information (counts, trends, mg/kg, soft hints) for the owner to discuss with their vet.

    All code, database fields, API routes, and identifiers must be in English.
    All user-facing UI text and the vet report export must be in Brazilian Portuguese (pt-BR).

    Mobile-first is a hard constraint: the app is used on a phone ~99% of the time. Primary actions
    must be thumb-reachable, tap targets large, and the critical "log a seizure" path must be fast
    and one-handed. Desktop is supported but secondary (centered, max-width layout).
  </overview>

  <core_features>

    <auth>
      - Single-user app. Login via email + password (NextAuth CredentialsProvider), JWT cookie session.
      - No public pages, no registration. The only user is seeded from ADMIN_EMAIL + ADMIN_PASSWORD.
      - Middleware protects every route except the auth/login route.
      - Login page at /login, redirect to / (home dashboard) after authentication.
    </auth>

    <home_dashboard>
      - Route: / — the landing page, shared by both streams. Mobile-first.
      - Leads with a large, thumb-reachable primary action: "Registrar crise" (log a seizure).
      - "Tempo desde a última crise" — a prominent counter (e.g. "12 dias sem crises").
      - Compact medication status strip: shows only meds that need attention (amber/red), each with
        its "repor até" date. If everything is OK, a calm "Estoque em dia" state.
      - Bottom tab navigation (mobile): Início · Crises · Medicamentos · Mais.
    </home_dashboard>

    <seizure_log>
      - Route: /crises
      - Fast logging path:
          * "Registrar crise" opens a minimal form pre-filled with the current date/time.
          * Minimum to save: occurred_at (defaults to now, editable) + type (Generalizada | Focal).
          * Everything else is optional and below the fold: duration, severity, rescue-given, notes.
          * Optional duration timer: "Iniciar cronômetro" to time the episode live; stop auto-fills
            duration_seconds. Duration can also be typed manually after the fact.
      - Auto-flags on save:
          * is_cluster = true if another episode exists within the prior 24h of occurred_at.
          * Episodes with duration > 5 minutes are visually flagged as emergency-level (status epilepticus
            risk). This is a display flag, not a separate field beyond duration.
      - Episode list: reverse-chronological, each row shows date/time, type, duration, severity,
        cluster/emergency badges. Tap to view/edit; edit and delete supported (log fast, annotate later).
    </seizure_log>

    <seizure_trends>
      - Route: /crises (dashboard section) — date-range filter: 3m | 6m | 12m | tudo.
      - Frequency over time: bar chart of seizures per week and per month (toggle).
      - Medication-change markers: vertical markers overlaid on the frequency chart, one per
        MedicationSchedule.effective_from, labeled with the med + change (e.g. "Fenobarbital ½ → 1").
        Lets the owner correlate a dose change with subsequent frequency.
      - Time since last seizure: big counter + longest seizure-free streak.
      - Breakdown: counts by type (generalized/focal), by severity, and by time-of-day; cluster events
        called out separately.
    </seizure_trends>

    <medication_management>
      - Route: /medicamentos
      - Medication list/dashboard: one card per active medication, color-coded by urgency:
          * OK (verde) — comfortably above the reorder window.
          * Reabastecer em breve (âmbar) — within the reorder lead-time window.
          * Urgente (vermelho) — at/below the lead-time window, or out of stock.
      - Each card shows: name, category, current stock (e.g. "28,5 comprimidos"), days remaining,
        "repor até" date, the active schedule (doses × units), and (if strength_mg set) computed mg/kg
        using the latest weight.
      - Create medication: name, category (continuous | otc | compounded), form (pill | capsule | tablet),
        strength_mg (optional), starting stock, schedule (dose times + units_per_dose, allowing fractions
        like 0.5), reorder_lead_time_days (defaulted by category: compounded 7, otc/continuous 3).
      - Edit medication metadata (name, category, form, strength_mg, lead time, active flag).
      - Per-medication actions:
          * "Repor estoque" — add a restock transaction (+N, with date).
          * "Corrigir estoque" — recount adjustment that snaps current stock to the actual counted number.
          * "Editar agendamento" — change dose times/units; writes a NEW MedicationSchedule row and closes
            the previous one (effective_to). This (a) adds a marker to the seizure timeline and
            (b) regenerates the "Adicionar ao Google Agenda" (.ics) action.
      - Days-remaining math:
          * daily_consumption = Σ(units_per_dose) across the active schedule's dose times.
          * days_remaining = floor(current_stock ÷ daily_consumption).
          * run_out_date = today + days_remaining.
          * reorder_by_date = run_out_date − reorder_lead_time_days.
      - Stock model (v1): the ledger stores only MANUAL rows — "restock" (+N) and "adjustment" (recount,
        sets stock to a counted number). Consumption is NOT stored as rows in v1; it is COMPUTED on read.
          * baseline = SUM(quantity of restock/adjustment rows with occurred_at ≤ now); the most recent
            "adjustment" acts as a recount checkpoint.
          * consumed_since = Σ over days from the last restock/adjustment to today of that day's
            daily_consumption (from whichever MedicationSchedule was active on each day).
          * current_stock = baseline − consumed_since (Decimal; fractional doses mean stock can sit at 28.5).
        This avoids a second cron and keeps recounts trivial. (v2's dose checklist may add literal
        "consumption" rows; the schema reserves the type for that.)
    </medication_management>

    <weight_and_dosing>
      - Route: /mais (or a sub-page) — simple weight log (WeightEntry: date, weight_kg) with a trend sparkline.
      - On medication cards with strength_mg set, show mg/kg = (units_per_dose_per_day × strength_mg) / latest_weight_kg.
      - Soft "possível subdosagem" hint when weight has risen enough that mg/kg dropped below a prior level.
        Informational only — never prescriptive, never an instruction to change a dose.
    </weight_and_dosing>

    <vet_report_export>
      - Route: /mais/relatorio — pick a date range, generate a pt-BR report containing:
          * Dog info + latest weight.
          * Seizure history table (date/time, type, duration, severity, cluster flag, notes) for the range.
          * Summary stats: total episodes, frequency (per month), longest seizure-free streak, type/severity breakdown.
          * Current medications + dose-change history within the range.
      - Output: print-friendly HTML (browser "Salvar como PDF") + a CSV export of episodes. Dependency-light,
        works on mobile.
    </vet_report_export>

    <google_calendar_integration>
      - When a medication schedule is set or changed, the app offers "Adicionar ao Google Agenda".
      - Generates an .ics file (and/or an "Add to Google Calendar" link) with one recurring VEVENT per dose
        time (RRULE: daily), e.g. 08:00 / 14:00 / 20:00. One tap to add to any calendar app.
      - No OAuth and no stored Google state. The app does not update/delete calendar events afterward;
        if the schedule changes, the UI reminds the owner to remove the old event and add the new one.
    </google_calendar_integration>

    <restock_alerts>
      - Dashboard: always-live color-coded status per medication (see medication_management).
      - Email digest: a Vercel Cron job runs once daily, recomputes each active med's status, and sends a
        single email (via Resend) to ALERT_EMAIL_TO listing only meds in amber/red, e.g.:
          "Composto manipulado: repor até sex 13/06 (7 dias restantes)".
        No email is sent on a day when nothing needs attention (no noise). The cron route is protected by CRON_SECRET.
    </restock_alerts>

  </core_features>

  <database_schema>
    <tables>
      <dogs>
        - id (uuid, PK)
        - name (text)
        - breed (text, nullable)
        - birthdate (date, nullable)
        - created_at, updated_at
        - (single row in practice, modeled as an entity to avoid hard-coding)
      </dogs>

      <weight_entries>
        - id (uuid, PK)
        - dog_id (uuid, FK dogs.id)
        - measured_on (date)
        - weight_kg (decimal(5,2))
        - created_at
      </weight_entries>

      <seizure_episodes>
        - id (uuid, PK)
        - dog_id (uuid, FK dogs.id)
        - occurred_at (timestamptz)
        - type (enum: generalized | focal)
        - duration_seconds (int, nullable)
        - severity (enum: mild | moderate | severe, nullable)
        - is_cluster (boolean, default false) — set automatically (another episode within prior 24h)
        - rescue_given (boolean, default false)
        - notes (text, nullable)
        - created_at, updated_at
      </seizure_episodes>

      <medications>
        - id (uuid, PK)
        - dog_id (uuid, FK dogs.id)
        - name (text)
        - category (enum: continuous | otc | compounded)
        - form (enum: pill | capsule | tablet)
        - strength_mg (decimal(8,2), nullable) — for mg/kg display
        - reorder_lead_time_days (int) — default by category (compounded 7, otc/continuous 3)
        - is_active (boolean, default true)
        - notes (text, nullable)
        - created_at, updated_at
      </medications>

      <medication_schedules>
        - id (uuid, PK)
        - medication_id (uuid, FK medications.id)
        - dose_times (text[] or jsonb) — list of HH:MM strings, e.g. ["08:00","14:00","20:00"]
        - units_per_dose (decimal(8,2)) — fractions allowed, e.g. 0.5
        - effective_from (date)
        - effective_to (date, nullable) — null = currently active; set when superseded
        - created_at
        - (history of schedule/dose changes — drives the timeline markers and the .ics)
      </medication_schedules>

      <stock_transactions>
        - id (uuid, PK)
        - medication_id (uuid, FK medications.id)
        - type (enum: restock | adjustment | consumption)
        - quantity (decimal(8,2)) — signed: restock +, consumption -, adjustment + or -
        - occurred_at (timestamptz)
        - note (text, nullable)
        - created_at
        - (v1 writes only restock/adjustment rows; current stock = baseline from these − consumption
          computed from the active schedule since the last row. The "consumption" type is reserved for
          v2's dose checklist, which may write literal consumption rows.)
      </stock_transactions>

      <!-- DEFERRED to v2 (not built in v1) — see implementation_steps / Linear: -->
      <dose_logs>
        - id (uuid, PK)
        - medication_id (uuid, FK medications.id)
        - scheduled_for (timestamptz)
        - taken_at (timestamptz, nullable)
        - status (enum: taken | skipped)
        - created_at
        - (powers the optional daily dose checklist; when present, refines that day's consumption)
      </dose_logs>
    </tables>
  </database_schema>

  <key_interactions>
    <logging_a_seizure>
      1. Tap "Registrar crise" on the home screen.
      2. occurred_at defaults to now (editable). Choose type (Generalizada | Focal).
      3. Tap Salvar — episode is recorded immediately (everything else optional).
      4. On save, is_cluster is set if another episode is within the prior 24h; duration > 5 min is flagged.
      5. Optionally reopen the episode later to add duration, severity, rescue-given, notes.
    </logging_a_seizure>

    <managing_stock>
      1. current_stock = baseline (SUM of manual restock/adjustment rows) − consumption computed from the
         active schedule since the last row. daily_consumption = Σ(units_per_dose) across the day's doses.
      2. days_remaining = floor(stock ÷ daily_consumption); reorder_by = (today + days_remaining) − lead_time_days.
      3. Card color: red if at/below lead-time window or out; amber if within the window; green otherwise.
      4. "Repor estoque" adds +N; "Corrigir estoque" snaps stock to a counted number via an adjustment row.
    </managing_stock>

    <changing_a_schedule>
      1. "Editar agendamento" on a medication.
      2. New dose times/units are saved as a new MedicationSchedule row; the previous row gets effective_to.
      3. A medication-change marker appears on the seizure-frequency timeline at effective_from.
      4. The "Adicionar ao Google Agenda" .ics regenerates; UI reminds the owner to remove the old calendar event.
    </changing_a_schedule>

    <daily_restock_digest>
      1. Vercel Cron calls GET /api/cron/restock-digest once daily (CRON_SECRET-protected).
      2. The route recomputes each active medication's status.
      3. If any are amber/red, send ONE Resend email to ALERT_EMAIL_TO listing them with "repor até" dates.
      4. If none need attention, send nothing.
    </daily_restock_digest>
  </key_interactions>
